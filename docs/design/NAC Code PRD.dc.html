1 · Summary

NAC Code is a graphical wrapper around agentic coding models. It replaces terminal-bound CLI harnesses with a familiar IDE layout: a left rail of chats grouped by workspace, a central conversation surface where the agent works, and a right inspector that surfaces live session state. It is built around three ideas existing tools handle poorly:

  • Static context as a managed library. Skills, agents, instructions, and files are first-class, searchable objects you attach to a chat — not pasted text. Attaching is budget-aware and reusable through saved configurations.
  • Everything is per-chat. Model, provider, agent, attached context, and compaction state belong to the individual chat, so switching conversations switches the entire working setup with it.
  • Visibility into work and cost. The agent's file changes are shown per repository — including repos outside the active workspace — viewable in-app or launchable in an IDE, and session cost is expressed in each provider's real billing unit.

2 · Problem & motivation

Current agent harnesses are overwhelmingly CLI-first. They make three recurring tasks painful:

  1. Context management is ad-hoc. Users paste files and instructions into prompts with no reuse, no inventory, and no sense of token cost until the window overflows. At scale — dozens of skills, many agents, standing instructions — a tray of chips above the input does not work.
  2. Sessions bleed together. Model choice, system instructions, and loaded context are global or per-invocation, not bound to a durable conversation. Returning to yesterday's task means rebuilding its setup.
  3. The work is invisible. Which files changed, in which repositories, on which branches — and what each turn cost — is buried in scrollback. There is no structured, reviewable record.

NAC Code addresses these directly with a managed context library, strict per-chat state, and dedicated review surfaces for changes and cost.

3 · Goals & non-goals

Goals

  • Make selecting, saving, and swapping static context fast and legible at the scale of 100+ skills, 50+ agents, and many instructions.
  • Bind a complete working configuration to each chat and make switching chats instantaneous and lossless.
  • Support multiple model providers behind one interface, with honest, provider-specific cost reporting.
  • Give a complete, structured view of the agent's file changes across all impacted git repositories, reviewable without leaving the app.
  • Keep power users on the keyboard via a command palette.

Non-goals (v1)

  • Replacing the IDE. NAC Code reviews and launches into IntelliJ / VS Code; it is not a full code editor.
  • Multi-user collaboration, shared sessions, or team permissioning.
  • Authoring/governance of skills and agents beyond browsing and attaching (creation is a stub in v1).
  • Mobile or web-responsive layouts — this is a desktop application with a fixed minimum width.

4 · Personas & use cases

Primary persona — the agent power user

An engineer who runs coding agents across several repositories and services daily, juggling many parallel tasks. Comfortable with CLIs but frustrated by their lack of organization. Cares about controlling cost and reviewing exactly what the agent did.

Representative use cases

  • Resume a task. Open yesterday's chat and have its model, agent, and context restored exactly.
  • Stand up a focused session. Apply an “Infra & Deploy” context configuration to a fresh chat in one click.
  • Branch from a long session. Compact a sprawling chat and spawn a clean one with the condensed context, keeping the original.
  • Review & ship. Inspect every changed file across the main repo and an out-of-workspace shared package, then open the shared package in IntelliJ to finish.
  • Control spend. Switch a routine chat to a local LM Studio model (free) and reserve Claude/GPT for hard problems, watching cost update live.

5 · Product principles

  • The chat is the unit of work. Configuration lives on the chat, never globally.
  • Context is inventory, not prose. It is browsed, searched, costed, and reused.
  • The inspector shows the present; dedicated views show history. Live state stays lightweight; stats and changes get their own room.
  • Honest cost. Each provider's spend is shown in its own unit — dollars, credits, or free.
  • Keyboard-first, mouse-complete. Every primary action is reachable from the palette and from the UI.

8 · Requirement relationships

The feature set has one backbone and several layered dependencies radiating from it.

The spine — FR-4.1 Per-chat state

Per-chat configuration is the single requirement the most others depend on. Attaching context (FR-5.5), applying a configuration (FR-6.2/6.3), choosing a provider/model (FR-7.4), selecting an agent (FR-8.1), branching (FR-9.3), and every inspector/stats readout (FR-10.4/10.5/10.6, FR-11.x) all read or write per-chat state. If FR-4.1 is correct, those features compose naturally; if it leaks to global state, all of them regress together.

Dependency chains

  • Context spine: FR-5.2 items → FR-5.5 attach → FR-5.7 budget → FR-6.1 configs → FR-6.2 picker → FR-10.6 summary. Configuration is sugar over the attach primitive.
  • Cost spine: FR-7.2 providers → FR-13.1 cost semantics → consumed by FR-10.4 (live) and FR-11.5 (breakdown).
  • Review spine: FR-3.3 inline diff → FR-12.1 changes view → FR-12.2 repo grouping → FR-12.6 full viewer + FR-12.4 IDE launch.
  • Compaction spine: FR-9.1 compact → FR-9.2 window reduction; FR-9.1 + FR-4.1 → FR-9.3 branch → FR-2.4 indicator.
  • Shell spine: FR-1.1 shell → FR-1.2 layouts → FR-10.9 docking; FR-1.3 rail and FR-14 palette are alternate entry points into routes.

Create and edit Skills, Agents, and Instructions inside the app, not just browse and attach. Planned.

Enforce workspace vs global scope, not merely display it: a workspace-scoped item is only attachable within its workspace. Planned.

12 · Glossary

  • Harness — the application layer that runs an agent against a model and a set of tools.
  • Static context — reusable, attachable knowledge objects (skills, agents, instructions, files) pinned to a chat.
  • Configuration — a named, reusable bundle of context items.
  • Compaction — condensing a chat's context to reduce window usage.
  • Branch / spawn — creating a new chat that inherits compacted context while preserving the original.
  • Workspace — a named grouping of related chats (typically per project/repo).
  • Inspector — the right panel of live, present-tense session state.
